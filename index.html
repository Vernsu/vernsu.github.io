发表于 2017-02-01

delegate 模式

Delegate 是 Cocoa 框架中的一种常见设计模式，这种模式有利于代码解耦和的，在iOS开发中经常被使用。

使用 delegate 一般分为4个步骤：

  1. 创建一个 delegate protocol
  2. 声明一个 delegate property
  3. 遵守并实现协议
  4. 调用 delegate

其中第二步，是新手经常犯错的地方。为了避免循环引用，在声明 delegate property 时，应该声明为 weak 引用。同时使用 weak 也使得 delegate 不存在其值时自动为nil，有效防止因为访问已经被回收的内存导致的 crash。

使用 Last-Modified 和 Etag 精确控制缓存失效时机

发表于 2017-01-31 | 分类于 网络缓存

《iOS 开发中常用的缓存策略：HTTP 协议篇》 一文中说到 Cache-control 含有 no-cache 时需要进一步验证缓存有效性。一般使用的验证机制有 Last-Modified 和 ETag。

Last-Modified

Last-Modified 在服务器返回的响应头中，标识资源的最后修改时间。相对应的客户端的请求头中有 If-Modified-Since ，标识客户端所记录的，资源的最后修改时间，值为最后一次请求时返回的 Last-Modified 字段的值。

第一次请求某一个URL时，服务器端的返回状态会是200，内容是你请求的资源，同时响应头中含有 Last-Modified ，格式类似这样：Last-Modified: Fri, 12 May 2006 18:53:33 GMT 。客户端第二次请求此 URL 时，根据 HTTP 协议的规定，客户端向服务器传送 If-Modified-Since 报头，格式类似这样： If-Modified-Since: Fri, 12 May 2006 18:53:33 GMT 。如果服务端的资源没有变化，则自动返回304状态码 Not Changed，内容为空，客户端就直接使用本地的缓存。当服务器端代码发生改变或者重启服务器时，则重新发出资源，返回和第一次请求时类似。这样就可以保证不向客户端重复发出资源，也保证当服务器有变化时，客户端能够得到最新的资源。

ETag

Etag 在服务器返回的响应头中，是资源的唯一标识符。相对应的客户端的请求头中有 If-None-Match ，标识客户端所记录的，资源的唯一标识符，值为最后一次请求时返回的 Etag 字段的值。

第一次请求某一个URL时，服务器端的返回状态会是200，内容是你请求的资源，同时响应头中含有 Etag ，格式类似这样：Etag："50b1c1d4f775c61:df3" 。客户端第二次请求此 URL 时，根据 HTTP 协议的规定，客户端向服务器传送 If-None-Match: 报头，格式类似这样： If-None-Match: "50b1c1d4f775c61:df3"。如果服务端的资源没有变化，则自动返回304状态码 Not Changed，与Last-Modified一样。

客户端处理

若缓存策略为默认的，iOS 系统会自动的处理 Last-Modified 和 ETag 并使用缓存。

如果想自己用代码处理 Last-Modified 和 ETag，那么缓存策略则必须使用 NSURLRequestReloadIgnoringCacheData，忽略缓存，每次都要向服务端进行校验。然后再用代码判断是否使用缓存。

另外如果更新频率较高，比如微博消息列表、一般的处理方式是添加一个接口，客户端将本地缓存的最后一条数据的的时间戳或 id 传给服务端，然后服务端返回新增的数据，没有新数据则返回 nil 或 304。

iOS 开发中常用的缓存策略：HTTP 协议篇

发表于 2017-01-28 | 分类于 网络缓存

《iOS 开发中常用的缓存策略：客户端篇》一文中我们提到 iOS 中 URL 加载系统默认的缓存策略使用当前 URL 的协议中的缓存策略。本文只关注 HTTP 协议中的缓存策略。

实际上 HTTP 协议中的缓存策略可以分为三部分：

  1. 缓存存储策略
  2. 缓存过期策略
  3. 缓存对比策略

使用 HTTP 头部中的 Cache-Control

使用 URLCache 的一个优势在于，系统可以自动的理解 HTTP 头部中的 Cache-Control 字段定义的缓存策略。我们只需要服务端根据 HTTP 协议设置 Cache-Control 的内容，URLCache 就知道, 这个缓存应该是什么时候过期了，客户端中的实现不需要写任何代码。

我们可以使用抓包工具或者在返回体中打印响应头信息:

Cache-Control 常见指令有：

  • max-age=86400 ：表示过期时间。缓存在一天内有效（60 秒 x 60 分 x 24 小时）。

  • no-store ： 表示不允许缓存响应，每个请求必须获取完整的响应。例如：一个包含个人隐私数据或银行数据的响应。每次用户请求该资源时，都会向服务器发送一个请求，每次都会下载完整的响应。

  • no-cache ： 表示必须先与服务器确认返回的响应是否被更改，然后才能使用该响应的缓存来满足后续对同一个网址的请求。因此，如果存在合适的验证方式 (ETag 或者 Last-Modified)，no-cache 会发起往返通信来验证缓存的响应，如果资源未被更改，则使用缓存。如果没有验证机制，则不使用缓存，直接进行网络请求。

关于 Cache-Control 的更多内容请参考 W3C的相关文档。

自定义缓存策略

只要有 URLCache 缓存了数据。如果你觉得服务端提供的缓存策略不合适，你可以指定 URLCache 自己的缓存策略。

URLRequest.CachePolicy.returnCacheDataDontLoad

提醒

  1. 最后必须提醒一下，我们在测试时经常会踩的坑：测试过程中服务器将响应的内容换了，我们不停的请求数据但是本地内容一直没有更新，因为服务端的缓存策略只给了max-age并没有标记为 no-cache，这个时候需要清理一下缓存。

  2. 如果一个请求的响应头中有 Transfer-Encoding: Chunked, 那 URLCache 也不会缓存。

iOS 开发中常用的缓存策略：客户端篇

发表于 2017-01-27 | 分类于 网络缓存

几种客户端缓存策略

我们在《iOS 开发中 URLCache 的应用》一文中讨论了 URLCache 的构建。

使用 URLCache 的最大好处在于可以指定缓存策略。缓存策略由请求（客户端）和响应（服务端）分别指定。几种客户端缓存策略：

reloadIgnoringLocalCacheData

忽略缓存，必须进行网络请求。

returnCacheDataElseLoad

尝试读取本地缓存，如果没有本地缓存数据，才会去请求网络。这个策略有一个重大的缺陷：无法控制缓存过期时间。

requestReturnCacheDataDontLoad

只读本地缓存，无论如何也不进行网络请求（也就是离线模式）。

useProtocolCachePolicy

默认的缓存策略，使用当前 URL 的协议中的缓存策略。

对于 HTTP 协议来说，URLCache 根据响应头来执行缓存策略。服务器可以在返回的响应头中加入 Cache-Control 字段来告诉客户端应该执行的缓存行为。具体的实现《iOS 开发中常用的缓存策略：HTTP篇》中会详细介绍。

CachePolicy 使用时机

可以在两个地方设置 CachePolicy ：

  • 为每个 URLRequest 设置单个请求的 cachePolicy:request.cachePolicy = .returnCacheDataDontLoad
  • 为每个 URLSessionConfiguration 设置 cachePolicy，该策略被 configuration 的 Session 下所有的请求共享:configuration.requestCachePolicy = .returnCacheDataDontLoad

另外需要注意，URLCache 只能缓存 GET 请求, 无法缓存POST请求。因为 response是通过 URL+Parameters 为 key 存储的。从数学的角度来讲，GET 的结果是 幂等 的，就好像字典里的 key 与 value 就是幂等的，而 POST 不幂等。从这个意义上说，一个文件的资源链接，也是 GET 请求。

介绍

在实际项目开发中，可能需要对网络请求进行缓存。例如网络图片，基本的缓存需求是：每张的图片 URL 都对应在本地保存的一个图片，这样当请求之前访问过的 URL 将直接从本地读取这张图片，为了效率更高，我们还会把图片保存在内存中。当开始请求一个图片 URL 时，我们先从内存中尝试读取这张图片，如果没有，再尝试从硬盘中读取这张图片，如果还是没有，才真的去网络上请求这个图片资源。

Apple 已经提供了一个强大的网络请求缓存类：URLCache。URLCache 提供了内存和硬盘缓存的综合缓存机制，能够控制内存和硬盘缓存的大小，并且可以控制缓存数据的存储路径。

Swift 中的 static 和 class 关键字

发表于 2017-01-25

static 和 class

在 Swift 中 Static 和 class 都是表示「类型范围作用域」的关键字。

在所有类型（class、struct、enum ）中使用 static 修饰都可以表示类方法类与属性（包括存储属性和计算属性）。

class 是专门用在 calss 类型中修饰类方法和类的计算属性（注意：无法使用 class 修饰存储属性）。

在 class 类型中 static 和 class 的区别

如此说来，在 class 类型中 static 和 class 都可以表示类型范围作用域，那他们之间有什么区别？

  1. class 无法修饰存储属性，而 static 可以。
  2. 使用 static 修饰的类方法和类属性无法在子类中重载。也就是说 static 修饰的类属性和类方法包含了 final 关键字的特性。

ststic 在 Protocol 中的使用

Swift 中 class，struct，enum 都可以实现某个指定 protocol。在 protocol定义一个类方法或者类计算属性，显而易见，应该使用 static 关键字来修饰。使用 protocol 时，在 struct 和 enum 中仍然使用 static，在 class 类型中 class 和 static 关键字都可以使用。

这种使用了和正常状态同类型的特殊值来表示失败，没有值，不存在的含义，我们通常将这种值就叫做哨兵值。

很多语言简单的处理了哨兵值，比如 Objective-C 中有 nil概念，但是仅仅只是和0的含义相同。

这就是哨兵值惹的祸，这样的问题防不胜防。

如果一种语言允许任何东西为 null，那么任何东西都可以是 null。换句话说，我们使用任何一个对象都冒着它并不存在的风险，而编译器对这种状态一无所知。
